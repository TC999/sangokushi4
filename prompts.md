# 项目：基于净室规格文档（在线源）—— 现代 GUI 重写 DOS 版《三国志4》

## 一、项目背景与原则

- 本项目已通过 **净室设计（Clean Room）** 完成逆向分析，生成了一套完整、独立于原始代码的**功能规格文档**（全部托管在指定的在线位置）。
- 这些文档只描述游戏的**行为、数据结构、算法流程、交互逻辑和界面布局**，不包含任何原始程序的汇编、伪代码或二进制信息。
- 当前为 **“重写”阶段**：需**严格且仅依据**这些在线规格文档，用现代编程语言重新实现一个功能完全等价的游戏程序。
- **铁律**：整个过程中，**AI 绝对禁止访问或引用任何原始 DOS 可执行文件、反汇编结果、IDA 数据库或逆向工程中间产物**。违规即终止。

## 二、核心目标

1. **功能等价**：重写后的程序逻辑、行为、数据流转必须与在线规格文档描述完全一致。
2. **现代桌面应用**：生成一个**独立的图形窗口程序**，双击可执行文件直接启动，**无需终端/命令行**。
3. **交互完整**：支持键盘和鼠标操作，交互方式与规格文档定义的原始游戏一致。
4. **全自动开发**：由 OpenCode Agent 全程自主驱动，自动抓取文档、规划、编码、测试。
5. **可追溯性**：每个模块、每个函数均需在注释中注明对应的在线规格文档章节或锚点。

## 三、技术规范

### 3.1 目标运行环境
- **目标平台**：Windows（优先），可兼顾 Linux/macOS（如采用跨平台框架）。
- **启动方式**：双击可执行文件 → 直接显示游戏主窗口，无控制台窗口。
- **窗口特性**：支持窗口缩放、关闭等标准操作（可根据规格决定是否全屏）。

### 3.2 目标语言与框架
- **语言**：推荐 Rust 或 C++17+，亦可选 Go / Python（但需确保有成熟的 GUI/游戏框架）。
- **推荐框架**（任选其一）：
  - **Rust**：`winit` + `pixels` / `egui` / `macroquad` / `SDL2` 绑定
  - **C++**：`SDL2`、`SFML`、`Qt`、`raylib`
  - **Python**：`pygame`、`PySide6`、`kivy`
  - **Go**：`ebiten`、`fyne`（若满足图形需求）
- **选择理由**：[请填写，如“Rust 保证内存安全且性能极高，SDL2 生态成熟跨平台”]

### 3.3 架构要求（分层）
- **平台抽象层（Platform Layer）**：
  - 封装窗口创建、事件循环（键盘/鼠标）、图形渲染（画点、画线、绘制精灵/字符）、声音（若有）、文件 I/O、随机数、时间等。
  - 该层是唯一与操作系统/框架打交道的部分，其余层完全平台无关。
- **核心逻辑层（Game Logic Layer）**：
  - 完全独立于平台，实现规格文档中的所有游戏机制（内政、战斗、外交、AI、存档等）。
  - 只与数据结构和纯算法打交道，不直接调用任何 GUI API。
- **表现层（Presentation Layer）**：
  - 使用平台抽象层提供的接口，将核心逻辑层的状态渲染为图形窗口内容。
  - 处理用户输入（键盘/鼠标事件），将其转换为对核心逻辑层的调用（如移动光标、选择菜单、点击按钮等）。
- **数据模型**：严格映射规格文档中定义的所有数据结构（武将、城市、部队、势力等），使用现代安全容器。

### 3.4 代码质量
- 每条关键逻辑必须注释引用规格文档的章节/编号。
- 单元测试覆盖率 ≥ 80%（核心逻辑层独立测试）。
- 遵循目标语言惯用法与最佳实践。

## 四、工作流（OpenCode Plan → Build 双模式）

### 阶段零（前置）：自动获取在线规格文档
1. Agent 启动后，使用内置网络工具（`curl`、`/web fetch` 或 Python `requests`）批量访问用户提供的在线文档 URL。
2. 将获取到的内容（HTML→纯文本/Markdown，PDF→文本提取）合并缓存为本地文件（如 `./specs/specs_combined.md`），作为唯一真理源。
3. 若文档需登录或动态加载，尝试寻找公开分享链接；若无法获取，立即请求用户提供静态导出版本，**不得自行臆测**。

### 阶段一：Plan 模式 —— 架构与 UI 规划
1. 深度阅读已缓存的规格文档，重点提取：
   - 游戏所有功能模块与数据流
   - 用户界面布局、菜单结构、交互方式（键盘快捷键、鼠标点击/拖拽等）
   - 画面尺寸、颜色模式（字符/图形）、刷新逻辑
2. 输出 **《重写架构设计文档》**，包括：
   - 模块划分与职责
   - 数据类型映射（规格 → 目标语言）
   - 平台抽象层接口设计（窗口、事件、渲染）
   - 关键算法流程（仅基于规格描述的算法）
   - UI 结构树（主界面、子界面、对话框）
   - 构建系统与依赖方案（CMake/Cargo等）
3. **反复迭代**，直至设计方案确认，方可进入 Build。

### 阶段二：Build 模式 —— 逐模块实现
按以下顺序逐步实现，每完成一个模块即运行测试验证：

1. **平台抽象层（基础）**：窗口创建、事件循环、基础图形绘制、键盘/鼠标输入。
2. **数据核心**：所有数据结构、序列化/反序列化（存档）。
3. **核心逻辑层**（模块化实现，顺序可调）：
   - 主循环与状态机（标题、菜单、游戏、战斗等）
   - 地图与移动
   - 内政系统
   - 军事与战斗
   - 外交系统
   - AI 决策
   - 存档/读档
4. **表现层**：将核心逻辑状态映射到屏幕元素，实现完整交互界面。

### 阶段三：集成与验证
1. 集成所有模块，编译为独立的桌面可执行文件。
2. 运行完整功能性测试，验证是否符合规格文档所有用例。
3. 测试时模拟键盘/鼠标操作，确保交互响应正确。
4. 输出测试报告，标注通过/失败项。

- **红线**：绝对禁止读取任何本地或网络上的原始 EXE、反汇编文件、IDA 输出等；一旦发现，立即忽略并报告。
