# 项目：基于净室规格文档（在线源）—— 现代语言重写 DOS 版《三国志4》

## 一、项目背景与原则

- 本项目已通过 **净室设计（Clean Room）** 完成逆向分析，生成了一套完整、独立于原始代码的**功能规格文档**。
- 这些文档**全部托管在指定的在线位置**（URL 列表），内容仅描述游戏的**行为、数据结构、算法流程、接口契约和逻辑规则**，不含任何原始程序的汇编、伪代码或二进制。
- 当前为 **“重写”阶段**：需**严格且仅依据**这些在线规格文档，用现代编程语言重新实现功能完全等价的游戏程序。
- **铁律**：整个过程中，**AI 绝对禁止访问或引用任何原始 DOS 可执行文件、反汇编结果、IDA 数据库或逆向工程中间产物**。违规即终止。

## 二、核心目标

1. **功能等价**：重写后的程序逻辑、行为、数据流转必须与在线规格文档描述完全一致。
2. **现代技术栈**：使用跨平台现代语言（如 Rust / C++17+ / Go / Python），架构清晰可维护。
3. **全自动运行**：由 OpenCode Agent 全程自主驱动，无需人工干预。
4. **可追溯验证**：每个模块、每个函数均需在注释中注明对应的在线规格文档章节或锚点。

## 三、技术规范

### 3.1 目标语言
选择：**Rust**
理由：**内存安全、高性能、跨平台**  

### 3.2 架构要求
- **分层架构**：
  - **平台抽象层**：封装文件系统、终端/图形输出、时间、随机数等，模拟 DOS 中断调用行为。
  - **核心逻辑层**：完全平台无关，实现规格中的所有游戏机制（内政、战斗、外交、AI、存档等）。
  - **表现层**：基于抽象接口实现文本或简易图形界面。
- **数据模型**：严格映射规格文档中定义的所有数据结构（武将、城市、部队、势力等）。
- **动态加载机制**：若原始有 overlay，采用现代动态库或模块化加载模拟。

### 3.3 代码质量
- 每条关键逻辑必须注释引用规格文档的章节/编号。
- 单元测试覆盖率 ≥ 80%。
- 遵循目标语言惯用法与最佳实践。

## 四、工作流（OpenCode Plan → Build 双模式）

### 阶段零（前置步骤）：自动获取在线规格文档
1. Agent 启动后，**首先**使用内置网络工具（如 `curl`、`/web fetch` 或 Python `requests`）批量访问用户提供的在线文档 URL 列表。
2. 将获取到的所有页面内容（HTML 需提取纯文本/Markdown；若为 PDF/Word，需尝试文本提取或提示用户提供纯文本版本）合并整理为本地缓存文件（如 `./specs/specs_combined.md`），作为后续唯一真理源。
3. **重要**：若文档需登录或动态加载，Agent 需尝试寻找公开分享链接；若无法获取，立即向用户请求静态导出版本，不得自行臆测。

### 阶段一：Plan 模式 —— 架构规划
1. 深度阅读已缓存的规格文档全文。
2. 输出 **《重写架构设计文档》**，包括：
   - 模块划分与职责
   - 数据类型映射（规格 → 目标语言）
   - 平台抽象层接口设计
   - 关键算法流程（仅基于规格描述的算法，禁引原始代码）
   - 构建系统与依赖方案
3. **反复迭代**，直至设计方案确认，方可进入 Build。

### 阶段二：Build 模式 —— 逐模块实现
按以下顺序逐步实现，每完成一个模块即运行测试验证：

1. **平台抽象层**：文件 I/O、随机数、时间、输出等。
2. **数据核心**：所有数据结构及内存管理。
3. **游戏引擎**：主循环、事件调度、状态机。
4. **功能模块**（建议顺序）：
   - 地图与移动
   - 内政系统
   - 军事与战斗
   - 外交系统
   - AI 决策
   - 存档/读档
5. **表现层**：用户交互界面。

### 阶段三：集成与验证
1. 全模块集成编译。
2. 运行完整功能性测试，验证是否符合规格文档所有用例。
3. 输出测试报告，标注通过/失败项。

## 五、OpenCode 交互与约束

- **获取文档**：使用 `/run curl -L <URL>` 或内置 fetch 工具下载在线文档；若为多页，需递归或遍历链接获取完整内容。
- **上下文管理**：若文档过大无法全量放入上下文，可使用 `/compact` 压缩已读内容，或采用检索增强（如将文档拆分嵌入本地向量库），但**核心知识必须源于在线规格**。
- **任务追踪**：每子任务前使用 TodoList 建立进度清单。
- **外部查询**：遇到现代语言等效实现问题，可使用 `/web search` 查询通用编程方案（但严禁搜索原始 DOS 代码或相关反汇编）。
- **自我审查**：每次 Build 后使用 `/review` 进行代码审查。
- **红线**：绝对禁止读取任何本地或在线路径下的原始 EXE、反汇编文件、IDA 输出等；若无意接触，立即忽略并报告。
